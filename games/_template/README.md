# games/_template

Skeleton for onboarding a new game. The `gameserver.yaml` here ships with the *rootless* hardening baseline **deferred by default** — a documented `nosemgrep` block stands in for the securityContext (`runAsNonRoot` / `readOnlyRootFilesystem` and friends) — because that's what **all 6 current games** do, not a minecraft-only exception. Most upstream game bases bootstrap as root (they chown a data dir to a configurable PUID/PGID, then drop privileges via gosu/su-exec before launching), and a forced non-root user or read-only root filesystem breaks that bootstrap. The manifest stays gate-clean via the `nosemgrep` block. Every game's image bakes in the grizzly supervisor as PID 1 (wrapping the game as its child), which owns the Agones SDK lifecycle and serves the in-pod control API the bot/Gary drive; `minecraft/`/`factorio/` are the worked examples. **If your game's base genuinely runs rootless** (no root chown on startup), it's one of the exceptions: delete the `nosemgrep` block and restore the full hardening baseline documented below.

## Onboarding a game

1. Copy this directory: `games/_template/` → `games/<game>/`.
2. Fill in `Dockerfile`: `REPLACE_BASE_IMAGE`/`REPLACE_DIGEST` (the game's own upstream image, pinned), `REPLACE_CHILD_CMD` (the command the supervisor launches), and the optional `SUPERVISOR_DATA_DIR`/`SUPERVISOR_RCON_PORT` env vars if they apply. See `games/minecraft/Dockerfile` for a filled-in example.
3. Replace every `REPLACE_*` placeholder in the manifests: `REPLACE_GAME` (name/selector/labels), `REPLACE_IMAGE` (the image this directory's `Dockerfile` builds — a tag or digest, never floating `:latest`), `REPLACE_PORT` / the `containerPort` / `nodePort` (a free port in **7000–7010**, the edge-forwarded band), the data `mountPath`, env, and resource sizing.
4. That's all the wiring — do **not** add `<game>` to `games/kustomization.yaml`. The catalog is all-dynamic: the bot renders each `games/<id>/` into a uniquely-named instance on demand per `/create`, so a new game is picked up automatically. `games/kustomization.yaml` is the always-on set (deliberately empty); adding a game there would run it 24/7. See `games/README.md`.
5. Validate against the live Agones webhook before pushing: `kubectl apply --dry-run=server -k games/<game>/`. That only checks manifest shape — it doesn't build the image or boot the game.
6. Actually try it: `scripts/push-game-image.sh <game> dev` (or `just game-push <game> dev`) builds and pushes the composite image to the in-cluster registry — the operative dev loop for iterating on a supervisor/game change, since game images aren't gate-signed/auto-deployed by CI yet. To drive a `/create` against the new game before anything goes live, run the bot off-cluster with `scripts/local-bot.sh start` (or `just bot-start`): it points `GAMESERVERS_CATALOG_DIR` at this repo's `games/` automatically (the compiled default is the in-container path), so the new `games/<game>/` is picked up with no extra config. See the root `CLAUDE.md`'s "Running & configuring the bot" section for both.

## The security baseline (the full hardening, deferred by default)

This is the *full* hardening baseline — the **restore target**, not what the template ships. The template ships it deferred (see above) because the current games' root-bootstrapping bases need it deferred; restore the pieces below only if your game's base genuinely runs rootless.

- **Pod `runAsNonRoot: true` + `runAsUser`/`runAsGroup` + `fsGroup`** — no root in the pod; `fsGroup` makes the data PVC writable by the runtime user. Set `runAsUser` to the UID the game image actually runs as (itzg/minecraft-server = 1000).
- **`seccompProfile: RuntimeDefault`** + container **`capabilities: drop [ALL]`** + **`allowPrivilegeEscalation: false`** — restricted-PSS posture; no privilege escalation, no extra kernel surface.
- **`readOnlyRootFilesystem: true`** — the game may only write to mounted volumes. When restored, pair it with a writable `/tmp` (emptyDir) plus the data PVC, and add an emptyDir for any other path the image writes to. The template defers this by default (root-bootstrapping bases write outside the data volume during their chown step); `games/minecraft`/`factorio`'s `gameserver.yaml` show the deferred shape.
- **Whole-CPU limits** — fractional CPU limits throttle and the gate flags them; size the limit to the game or omit it to allow bursting.
- **The supervisor, not a readiness sidecar** — earlier revisions of this template bridged a game to the Agones lifecycle with a separate `agones-ready` busybox sidecar. That's been replaced everywhere: the supervisor is baked into the game image itself (this directory's `Dockerfile`), because a separate sidecar can *signal* the game process but can't *relaunch* it — see `docs/design/01-sidecar-agent-interface.md`.

## Notes

- **Data directory for the ops agent (`SUPERVISOR_DATA_DIR`)**: the supervisor scopes Gary's file tools (`browse_files`/`read_file`/`write_file`/`restore_file`) to this directory, defaulting to `/data`. If your game's data PVC `mountPath` is anything other than `/data`, set `SUPERVISOR_DATA_DIR` to match in the `Dockerfile`'s `ENV` — so the agent can reach the game's config and logs. Paths outside this directory (absolute paths, `..`) are refused.
- **RCON / console access**: only wire `SUPERVISOR_RCON_PORT` (and the matching supervisor dialect flag) if the game speaks it and you want Gary's `send_command` tool to issue console commands. See `games/minecraft/Dockerfile` for the worked example, including ephemeral password minting so no secret is baked into the image or any Kubernetes object.
- **Game port (`SUPERVISOR_GAME_PORT`)**: set this to the TCP port the readiness probe below should connect to. It defaults to `25565` (Minecraft's port) if left unset — every game other than Minecraft (whose port happens to be 25565) must set it explicitly, or the probe silently watches the wrong port. `crates/bot/src/agones/tests/ports.rs`'s `real_game_ports_agree_with_their_readiness_probe` test catches a mismatch between this and the manifest for every real game, so a new game should keep that test green too.
- **Readiness (`SUPERVISOR_READY_LOG_PATTERN`)**: the supervisor only calls Agones `/ready` once the game is actually up, and by default proves that with a TCP connect to `SUPERVISOR_GAME_PORT`. That works for TCP games (Terraria) and for UDP games whose RCON/query port opens on boot (point `SUPERVISOR_GAME_PORT` at that TCP port). A UDP-only game that opens **no** TCP port (Valheim) would never satisfy the connect probe, so set `SUPERVISOR_READY_LOG_PATTERN` to a substring of the game's "server started" log line instead — readiness then fires on the first captured line containing it. Leave it unset to keep the TCP probe.
- **UDP games** (Valheim, etc.): set `protocol: UDP` on the port and Service. The edge forwards UDP+TCP, but TCP MSS clamping doesn't help UDP — a game sending >~1380-byte datagrams needs its own MTU/payload setting (the `wg0` tunnel is 1420). See `docs/activation-status.md`.
- **Multiple advertised ports**: most games need only one leased `NodePort` (the default "remap" path — nothing to configure). If your game's client dials more than one server-advertised port (like Satisfactory's game + Reliable Messaging ports), see `games/satisfactory/service.yaml` for the worked example of the `advertised-ports`/`friend-facing-port`/`port-env.*` annotations that put a game on that path.
- Promote a single `GameServer` to a `Fleet` + shim-managed per-instance NodePort Services once the ops agent/shim owns allocation.
