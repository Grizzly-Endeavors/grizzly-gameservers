# Activation Status — what's live

The infrastructure and the application are both up: the cluster runs Agones, the guardrails, and the all-dynamic game catalog (servers are provisioned per `/create`, none always-on), and the Discord bot + ops agent (Gary) is deployed and gate-signed via CI.

## Live (verified)

- **Edge forwarding** — Hetzner VPS + R730xd forward the **7000–7010** UDP+TCP range over `wg0` to the cluster (`grizzly-platform` ansible; ADR-019). Validated end-to-end: `<vps-public-ip>:7000` reaches the Minecraft pod and a real client joins. **Requires TCP MSS clamping** on the `wg0` forward path (`ingress-tunnel` role `wg0.conf` `PostUp`) — game traffic is L3-forwarded, so without clamping to the 1420 tunnel MTU, login succeeds but world-load packets black-hole. UDP games (Factorio, Palworld, Valheim, Satisfactory — all in the catalog) can't use MSS clamping and need their own MTU handling.
- **Agones `1.58.0`** — installed as its own Flux `HelmRelease` (`cluster/agones/`), operator + CRDs Ready in `agones-system`. Watches `game-servers`; its own hostPort range pinned to `8000–8100`, clear of the NodePort band. See ADR-001.
- **Guardrails** (`cluster/guardrails/`) — `game-servers` namespace, the `discord-bot` ServiceAccount + scoped Role/RoleBinding (also the ops agent's identity — Gary runs inside the bot process), and the `CiliumNetworkPolicy` egress leash. Leash verified: from a game pod, LAN/platform services (OpenBao `10.0.0.200:8200`) are blocked while DNS, the Kubernetes API, and the internet work.
- **Game catalog** (`games/`) — every `games/<id>/` directory is an on-demand template (Agones `GameServer` + NodePort `Service` + world PVC) the bot renders into a uniquely-named instance on a leased `7000–7010` NodePort per `/create`. Instances **land on a worker, not the edge node** — kube-proxy routes the static edge to the pod wherever it runs (no node-pinning). See ADR-002. The always-on `minecraft` server that validated bring-up was decommissioned once dynamic provisioning was proven; `games/kustomization.yaml` (the Flux-rendered always-on set) is now empty — the model is all-dynamic.
- **Flux wiring** (`grizzly-platform`) — `grizzly-gameservers-cluster` (path `./cluster`, `wait: true`) and `grizzly-gameservers-games` (path `./games`, `dependsOn` cluster), both reconciled from the `grizzly-gameservers` GitRepository on `main`. The app `HelmRelease` renders `./deploy`, carrying the OpenBao-backed `ExternalSecret`s (`discord-bot`, `ollama-api`, `grizzly-gameservers-db`, `grizzly-gameservers-s3`, `grizzly-gameservers-ingame`), the bot/agent Deployment, and the ClusterIP Service for the in-game agent endpoint.
- **NodePort range** — apiserver `--service-node-port-range` widened to `7000-32767` (IaC in `grizzly-platform` kubeadm-config; live + kubeadm ConfigMap reconciled) so game Services bind `7000–7010` 1:1 with the edge.
- **Discord bot + ops agent (Gary)** — `grizzly-gameservers-bot` Deployment running (1/1 Ready) on a gate-signed, digest-pinned image (`deploy/templates/deployment.yaml`).
- **CI gate** (`.github/workflows/deploy.yml`) — live: build → centrally-owned gate cosign-signs the digest on a clean pass → pins it into `deploy/values.yaml` → Flux reconciles. An unsigned image never reaches the cluster.

## Deferred

- **Full-namespace gate enforcement** — `game-servers` already carries `grizzly.io/gated=true`, but enforcement is bot-scoped only (ADR-003); the Agones SDK sidecar needs no carve-out (it's a `gcr.io` image the policy glob never matches), but game/supervisor images aren't yet gate-signed in CI, so full-namespace enforcement stays a documented follow-up (see `cluster/kyverno/README.md`).
- **Fleet-backed instances** — each `/create` currently renders its own standalone `GameServer`; promoting those to a `Fleet` (pre-warmed replicas for faster allocation) is a deferred optimization.
