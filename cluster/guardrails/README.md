# cluster/guardrails

The ops agent's leash. Lives in this repo (co-located with the agent code) so the code and what it's allowed to touch move in one reviewed, gated PR. See `docs/design/00-overview.md` → "Ops-agent guardrails".

- `namespace.yaml` — the `game-servers` `Namespace`. Defined here (not in `grizzly-platform`) so it exists under the same `./cluster` Flux Kustomization that installs Agones, which provisions per-namespace SDK RBAC into it. Labelled `grizzly.io/gated=true`, but enforcement is scoped to the bot image only — see `cluster/kyverno/README.md` for what that does and doesn't cover.
- `networkpolicy.yaml` — `CiliumNetworkPolicy` egress leash (the guardrail that matters most). Restricts egress to DNS + the Kubernetes API server + the public internet, denying private RFC1918 space so a compromised or prompt-injected game server / agent can't pivot into OpenBao or other platform services. Ingress is left open (internet-facing game servers behind the NodePort edge). The `bot-to-*-egress` / `game-to-*-egress` files below are narrow additive carve-outs on top of this leash — read them for the exact paths that are widened.
- `bot-rbac.yaml` — the `discord-bot` `ServiceAccount` + namespace-scoped `Role`/`RoleBinding`: create/delete `gameservers`, their NodePort `Service`s and PVCs, and read pod IPs to reach the in-pod supervisor control API. No patch, no exec. This is also the ops agent's ("Gary") identity — Gary runs inside the bot process and operates servers through the supervisor's in-pod control API, not through K8s exec/patch, so it never needed a separate ServiceAccount.
- `bot-to-supervisor-egress.yaml` — `CiliumNetworkPolicy` carve-out that widens the bot's otherwise-default-deny egress by exactly one path: bot → game-server pods, TCP `9359` (the supervisor control port), so `/stop`, `/start` and `/restart` can reach it.
- `bot-to-postgres-egress.yaml` — carve-out for the bot's durable state: bot → `10.0.0.200/32`, TCP `5432` (foundation Postgres, the home-channel + guild-config registries). Bot-only, and deliberately not the whole host — OpenBao on `8200` stays unreachable.
- `bot-to-s3-egress.yaml` — carve-out for world backups/archives: bot → `10.0.0.200/32`, TCP `7072` (the self-hosted versitygw s3-bulk gateway). Bot-only; the untrusted game pods never hold S3 credentials and stay fully leashed.
- `game-to-bot-agent-egress.yaml` — the reverse of `bot-to-supervisor-egress`: game-server pods → the bot workload, TCP `9360` (the in-game `@Gary` agent endpoint). Nothing else opens — game pods still can't reach each other, and `9360` is never exposed through the NodePort edge.

Cilium default-deny drops are silent: a blocked call just looks like a connect timeout from the caller's side, with nothing in the app's own logs pointing at the leash. Before chasing that as an app bug, confirm it with Hubble: `hubble observe -n game-servers --verdict DROPPED` (add `--since 5m` to narrow the window). A `DROPPED` verdict naming the destination IP/port means `game-servers-egress-leash` (or one of the carve-outs above) is the cause, not the app.
